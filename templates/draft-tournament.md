# Draft Tournament Template

Use for `variants` mode and for standard/deep substantive writing.

## Candidate generation

Generate genuinely different candidates. Do not produce cosmetic rewrites.

Suggested variant set:
- Plainspoken
- Warm/personal
- Sharp/opinionated
- Narrative/example-led
- Concise/professional
- Quiet/literary or contrarian, if appropriate

Variant count:
- quick: skip tournament or produce 2 mini-options
- standard: 3 variants
- deep: 5–6 variants

## Candidate review

For each candidate, assess:
- strongest opening
- strongest claim
- best example/detail
- best phrase
- best structure
- best ending
- weakest AI-sounding part
- what to steal

Optional scoring table:

```markdown
| Draft | Specificity | Voice | Clarity | Memorability | AI-slop risk | Audience fit | Notes |
|---|---:|---:|---:|---:|---:|---:|---|
| A | /5 | /5 | /5 | /5 | /5 | /5 | |
```

For AI-slop risk, lower is better.

## Best-parts harvest

Do not select one winner wholesale unless it is clearly best. Treat candidates as a parts bin.

```markdown
## Best parts harvested
- Opening: Draft [X] — [why]
- Structure: Draft [X] — [why]
- Sharpest claim: Draft [X] — [why]
- Best detail/example: Draft [X] — [why]
- Best phrase: Draft [X] — [why]
- Ending: Draft [X] — [why]
```

## Composite

Assemble a new draft stronger than any individual candidate.

## Final audit

Before presenting the final, check:
- concrete detail
- point of view
- rhythm variation
- excessive polish
- generic transitions
- over-neat ending
- factual or personal details not provided by user

## Output shape

```markdown
## Best parts harvested
- ...

## Final composite
...

## Notes
- ...

## Optional alternate
...
```

For concise requests, omit candidate drafts and show only the harvest/composite unless the user asked to see all variants.
